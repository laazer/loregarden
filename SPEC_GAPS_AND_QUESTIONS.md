## Overview

This document captures assumptions made during test design and identifies areas where the implementation specification may need clarification. These questions should be addressed during the IMPLEMENTATION_BACKEND stage.

## Assumptions Made in Test Design

### 1. Response Format

**Assumption:** Claude API response contains a JSON object with a `hierarchy` field containing an array of work items.

```json
{
  "hierarchy": [
    {
      "external_id": "...",
      "title": "...",
      "work_item_type": "...",
      "description": "...",
      "acceptance_criteria": [...],
      "priority": 3,
      "children": [...]
    }
  ]
}
```

**Question:** 
- Should the response format be different?
- Are there any other fields Claude should return (e.g., reasoning, confidence scores, alternative proposals)?
- Should the response include any metadata (tokens used, model info, generation timestamp)?

### 2. Prompt Template

**Assumption:** A single prompt is sent to Claude containing:
- Ticket title
- Ticket description
- Acceptance criteria
- Instructions for hierarchy generation format and constraints

**Question:**
- Should the prompt include additional context (e.g., existing workspace structure, prior decompositions)?
- Should there be different prompt templates for different ticket types (milestone vs feature vs capability)?
- Should the prompt be configurable/versioned?

### 3. Token Limit Handling

**Assumption:** When approaching token limits:
- The service returns whatever hierarchy has been generated so far
- Truncated hierarchies are still valid proposals
- No special indicator is needed (test just handles empty response)

**Question:**
- Should truncated proposals be rejected entirely?
- Should the service attempt to re-request with a shorter prompt?
- Should there be a maximum token budget per request?
- How should very large ticket descriptions be handled (truncate before sending to Claude)?

### 4. API Error Handling

**Assumption:** API errors (timeout, rate limit, auth failure) should:
- Return an empty hierarchy or error indicator
- Not throw exceptions that halt the system
- Allow retry logic at a higher level

**Question:**
- Should the service implement retry logic internally (with exponential backoff)?
- Should errors be logged and reported somewhere?
- Are there specific HTTP status codes that should trigger different behaviors?
- Should the service queue failed requests for later retry?

### 5. Hierarchy Validation

**Assumption:** Generated hierarchies must respect VALID_HIERARCHY rules:
- Milestone can contain: Feature, Bug
- Feature can contain: Capability, Bug
- Capability can contain: Task, Bug
- Task can contain: nothing (leaf)
- Bug can contain: nothing (leaf)

**Question:**
- Should invalid hierarchies be rejected completely?
- Should the service attempt to "fix" invalid hierarchies?
- Should there be a fallback behavior if Claude generates an invalid structure?

### 6. Determinism & Caching

**Assumption:** Same ticket input may produce different proposals on different calls (due to Claude's sampling).

**Question:**
- Should proposals be cached per ticket?
- Should there be a way to pin a specific proposal version?
- Should the service accept a `seed` parameter for deterministic generation?
- How long should cached proposals be kept?

### 7. Work Item Types & Levels

**Assumption:** Decomposition should generate hierarchies using the 5 defined types:
- Milestone
- Feature
- Capability
- Task
- Bug

**Question:**
- Should the service enforce that hierarchies always start with Milestone?
- Can a Feature be a root item (no Milestone parent)?
- Should there be limits on hierarchy depth?
- Should there be limits on number of children per item?

### 8. External ID Generation

**Assumption:** External IDs in the response are generated by Claude (based on the ticket being decomposed).

**Question:**
- Should external IDs follow a specific format/pattern?
- Should they be validated before insertion into the database?
- Should there be conflict detection if an external_id already exists in the workspace?
- Should the service append a suffix to avoid conflicts (e.g., `-1`, `-2`)?

### 9. Acceptance Criteria & Descriptions

**Assumption:** Claude should generate meaningful acceptance criteria and descriptions for each hierarchy item.

**Question:**
- Should these be validated for quality/length?
- Are there minimum/maximum lengths expected?
- Should empty descriptions/criteria be rejected or accepted?
- Should there be a maximum number of acceptance criteria per item?

### 10. Priority Assignment

**Assumption:** Claude assigns priority values in range [1, 3] (high to low).

**Question:**
- Should all items inherit parent priority or can they differ?
- Should there be rules about relative priority between siblings?
- What is the default priority if not specified?

## Questions for Specification Agent / Backend Implementer

### Configuration & Customization

1. **Model Selection:** Should the service use a specific Claude model version, or should it be configurable?
2. **Prompt Versioning:** Should there be multiple prompt templates for A/B testing or improvement iteration?
3. **Rate Limiting:** What are the expected request rates? Should there be per-workspace or per-user limits?

### Integration Points

4. **Database:** Should decomposition results be stored immediately or returned for approval first?
5. **API Endpoint:** Should this be a REST endpoint, or part of a larger ticket studio workflow?
6. **User Interaction:** Can users regenerate proposals? Can they edit/refine proposals before finalization?

### Failure Scenarios

7. **Partial Failures:** If some items in a hierarchy fail validation, should the entire proposal be rejected?
8. **Empty Response:** How should the system handle a completely empty response from Claude?
9. **Malformed Response:** Should the service attempt recovery or report failure immediately?

### Performance & Scale

10. **Caching:** Should decompositions be cached? For how long?
11. **Concurrency:** Can multiple users request decompositions simultaneously?
12. **Timeouts:** What is the maximum acceptable wait time for a decomposition request?

### Acceptance Criteria for Implementation

13. **Testing:** Should the implementation include its own test cases that use real Claude API?
14. **Monitoring:** Should there be metrics/logging for decomposition quality or error rates?
15. **Documentation:** What examples or use cases should be documented?

## Test Specification Questions

### Tests That Assume Behavior

The following test patterns make assumptions that should be verified:

1. **JSON Parsing Robustness**
   - Test assumes graceful handling of missing fields
   - Implementation may require strict validation instead

2. **Hierarchy Structure Validation**
   - Tests verify valid type progressions
   - Implementation needs to decide: validate/reject vs. auto-fix vs. warn

3. **Error Handling Patterns**
   - Tests expect empty array for timeout/errors
   - Implementation may need different error signaling

4. **Uniqueness Constraints**
   - Tests verify external_id uniqueness within response
   - Implementation needs cross-workspace uniqueness strategy

### Mock Data Assumptions

The test suite uses mock Claude responses. When implementing:

5. Should verify mock response format matches actual Claude API format
6. Should test with real API responses (at least in integration tests)
7. Should handle any API response variations

## Recommendations for Backend Implementer

1. **Start with a simple implementation** that returns a single hardcoded decomposition for test tickets, to verify the end-to-end flow works.

2. **Add real Claude API integration** incrementally, starting with small tickets and gradually increasing complexity.

3. **Implement comprehensive logging** of:
   - Input ticket content
   - Prompt sent to Claude
   - Claude's full response (before parsing)
   - Any parsing errors or validation failures

4. **Consider rate limiting** early, as Claude API has usage limits and costs scale with token count.

5. **Plan for future enhancements**:
   - Multiple decomposition proposals (let users choose)
   - User feedback/refinement loop
   - Decomposition templates by ticket type
   - Caching/versioning of proposals
